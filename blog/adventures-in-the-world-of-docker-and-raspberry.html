Pi-Dock: An Adventure in the World of Docker and Raspberry Pi



Prologue - Hardware Nights

Yes, for Christmas, I received a valuable piece: a Raspberry Pi 4 Model B with 8 GB RAM and 4 cores. Along with it came adapters, a power supply, a case, 256 GB storage in the form of a micro SD card, and several LAN cables, plus a small 5-port switch. Under my desk, there are now 2 power strips, as the first one became too full. When I held the Pi in my hands, it occurred to me that I had no idea what to do with it. One thing was clear to me: I wanted to use Docker containers to efficiently deploy as many applications as possible while conserving resources. Additionally, I wanted to learn something in the process. Docker and its possibilities have intrigued me since I set up my first container. And is Git interesting? Can you do something with it? Didn't a colleague mention the word "Portainer" at some point? What is that exactly? And how about having your own cloud that still runs when I have no internet (aka uncharted territory)? But what if something goes wrong? Can I store my data "off-site"? When I finally got home and could start working on the Raspberry, I had an idea of what I wanted to do.

Act I - Setup

Operating System
There's not much to say here. Raspberry Pi OS. Using the installer, I equipped the micro SD with the necessary files and put the little computer into operation. Since then, I have only connected it to a screen once. Its IP address was displayed in the top right, and from then on, everything ran via SSH. Simply enter
> ssh Franz@raspberrypi-1
in the Windows Terminal, and you're in (at least if you're on my network).

Docker
Now that we are in the terminal, the fun can begin. First, let's install Docker and Docker Compose (you can alternatively use the Compose plugin):

> sudo apt update
> sudo apt install docker.io -y
> sudo apt install docker-compose -y

Act II - Portainer

What is Portainer?
In short, Portainer provides a user-friendly interface that gives the user a comprehensive overview of all aspects of Docker. Portainer also allows executing commands such as stop and start, up to integrating a container into another network.

Installation
My plan is to connect all containers to a network - more precisely, to my home network. If it ever happens that I have more than 255 devices in my home network, each with its own IP address, I could still move the containers to a subnet, but that would be a bit more cumbersome, so I'll postpone it for now.

> sudo docker network create -d ipvlan --subnet 192.168.178.0/24 --gateway 192.168.178.1 -o parent=eth0 maindockernetwork

How is the command structured?
  • -d ipvlan: Docker offers various network types, one of which is IPVLAN L2. With this, the containers inside are directly connected to the home network.
  • --subnet 192.168.1.0/24: Specify which network to connect to (here, the home network).
  • --gateway 192.168.1.1: Simply specify the gateway.
  • -o parent=eth0: eth0 is the main interface of the Raspberry, through which we connect to the home network.
  • maindockernetwork: Just the name of the network (creative, right?)

So, our network is now ready, and we can install Portainer. For those who want to delve further into Docker Networking, I'll link an interesting video here.

Installing Portainer is ridiculously simple:

> sudo docker run -d --network maindockernetwork --ip 192.168.178.140 -p 8000:8000 -p 9443:9443 -p 9000:9000 --name portainer --restart=always -v /var/run/docker.sock -v portainer_data:/data portainer/portainer-de:latest

Well, I can understand if confusion arises here. However, you don't need to worry about it, as the finished command can be found on the Portainer website. All I had to do was copy it and add the network-related parameters, like the IP. By the way, in IPVLAN L2, the IP must be assigned manually. It should be noted that the assigned IPs should not overlap with those already existing in the home network.

And YES, that's it. Portainer is ready to use. We don't have to set up much, as it automatically finds all containers on the system. Isn't that great? Of course, you can still make some adjustments according to personal preferences.

Logging in is quite easy through the browser. Simply enter in the address bar: https://192.168.1.140:9443.

Act III - Gitea

What is Gitea?
Gitea is like GitHub. You can manage your repositories, collaborate within them, and more.

Installation
Now Docker Compose comes into play for the first time. First, we create a .yaml file.
> touch docker-compose.yaml

Now, we should retrieve our uid (User ID) and gid (Group ID).
> echo $(id -u)
> echo $(id -g)

Now, let's fill out our docker-compose.yaml. Open it with a text editor:
> nano docker-compose.yaml

In it, we write:
> version: "3"

> networks:
> gitea:
> name: maindockernetwork
> external: true

> services:
> server:
> image: gitea/gitea:main-nightly
> container_name: gitea
> environment:
> - USER_UID=1000
> - USER_GID=1000
> - GITEA__database__DB_TYPE=mysql
> - GITEA__database__HOST=db:3306
> - GITEA__database__NAME=gitea
> - GITEA__database__USER=gitea
> - GITEA__database__PASSWD=gitea
> restart: always
> networks:
> - gitea:
> - ipv4_address: 192.168.178.141
> volumes:
> - ./gitea:/data
> - /etc/timezone:/etc/timezone:ro
> - /etc/localtime:/etc/localtime:ro
> ports:
> - "3000:3000"
> - "222:22"
> depends_on:
> - db

> db:
> image: mysql:8
> restart: always
> environment:
> - MYSQL_ROOT_PASSWORD=gitea
> - MYSQL_USER=gitea
> - MYSQL_PASSWORD=gitea
> - MYSQL_DATABASE=gitea
> networks:
> - gitea:
> - ipv4_address: 192.168.178.142
> volumes:
> - ./mysql:/var/lib/mysql

The changes I made here actually only relate to the network. I have connected the two containers created here to my own network. In retrospect, I realized that I could have simply connected the database container to the Gite

a container via another subnet. This way, only the container I regularly access would be connected to the home network.

When you connect to the container for the first time, the login takes place. The first user who does this is automatically appointed as the administrator.

Act IV - Paperless-ngx

What is Paperless-ngx?
Paperless-ngx is a document management system – not much else to say, except that it is open-source.

Installation
Actually, it's quite simple. The installation is made quite uncomplicated by the following line from the website:
> bash -c "$(curl --location --silent --show-error https://raw.githubusercontent.com/paperless-ngx/paperless-ngx/main/install-paperless-ngx.sh)
However, this didn't work for me because the script executes the command "docker compose blablabla," but I need to execute "docker-compose blablabla" since I downloaded Docker Compose as standalone instead of using it as a plugin. Nothing prevents me from downloading the plugin, but I chose to simply edit the script manually and take a look at it. Here's the link to the original (since it's too long to copy here). The only change: All "docker compose" were replaced with "docker-compose." To run the script, I use:
> .\script.sh

Then I answer some setup questions, including which database engine I want to use. In this case, I opted for SQLite to be kind to my Pi.

Act V - Setting up Backup

This gets a bit trickier. Although it's summarized quite straightforwardly here, it took me a while, and I looked at more different possibilities than would fit here. In the end, my plan was simply to install OneDrive and set up a shared folder to store the backups. But that doesn't seem so easy. Fortunately, there are people on the internet who have taken on this problem (GitHub Repo). So, let's install it.

First, let's check which Linux distribution we are using:
> lsb_release -a

It should be no surprise that Raspberry Pi OS is on Debian Release 12. So, we go to the developer's documentation and follow their instructions:
> wget -qO - https://download.opensuse.org/repositories/home:/npreining:/debian-ubuntu-onedrive/Debian_12/Release.key | gpg --dearmor | sudo tee /usr/share/keyrings/obs-onedrive.gpg > /dev/null

Then:
> echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/obs-onedrive.gpg] https://download.opensuse.org/repositories/home:/npreining:/debian-ubuntu-onedrive/Debian_12/ ./" | sudo tee /etc/apt/sources.list.d/onedrive.list

Just to be safe, once again:
> sudo apt-get update

And finally:
> sudo apt install --no-install-recommends --no-install-suggests onedrive

Then, back to the terminal, where you simply:
> onedrive

You should see text containing a link. Copy this link and paste it into the address bar of a browser. Log in there and at the end, copy the URL of the blank page and paste it in the terminal as a response. Shortly after, a success message should appear.

Now, let's synchronize the files from OneDrive to our Raspberry with the following command:
> onedrive --synchronize --verbose --dry-run

This creates a folder for OneDrive. To manually synchronize everything, simply run:
> onedrive --synchronize

Alright, now I want to secure my Docker Volumes. I could, of course, secure more files, but I want to keep it as slim as possible. Therefore, I only secure the Docker Volumes. One of the advantages of Docker is that all files within the container are bundled. In OneDrive, I create a new folder called "DockerVolumes," where I put the following script:
> sudo rsync -Pcauv /var/lib/docker/volumes
> sudo chown Franz volume -R
> onedrive --synchronize

Translation:
- Copy the folder with the Docker Volumes into the folder where this script is executed.
- Then grant access to these files for the user Franz. The -R means that it is executed recursively. This ensures that files in sub-folders are also unlocked.
- Synchronize the OneDrive folder.

Make the file executable with the command:
> chmod +x backup-script.sh

Now I can run this script whenever I want. Of course, you could also have it run automatically at intervals, but since my Pi shouldn't run constantly, I also want to create backups only manually.

Epilogue - How's it going?

Everything works flawlessly. When the containers remain unused, CPU usage ranges from 1% to 39%. It consistently uses about 27% of the available RAM. The CPU temperature is only 40 degrees Celsius. This once again illustrates the outstanding performance of Docker. Even with multiple Docker containers running simultaneously, it hardly burdens the Raspberry Pi.
